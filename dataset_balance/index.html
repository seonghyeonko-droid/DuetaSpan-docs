B. 다양성 상한 — 어디가 병목인가

MCP 계열의 76k는 내재적 상한(문제 아님). 도구 카탈로그가 125개로 고정이고, 시나리오 76,251개는 공식 QA 데이터셋(SGD 파생)의 서수 커버리지 그대로다. 즉 벤치마크 정합성을 위해 의도적으로 공식 시나리오에 묶인 것 — 무한히 늘릴 대상이 아니라 커버리지를 다 쓰는 게 목표. 풀을 76k 이상으로 키우면 MCP는 시나리오를 재사용(시드로 표현만 달라짐)하지만, 이는 툴·벤치마크 특성상 자연스러운 것.

Memory · Persona는 우리가 자유롭게 키우는 kind. 공식 데이터셋에 묶이지 않은 자체 설계라, 오늘 Context DB 다양성 소스를 대폭 확장:

  • Memory: 카테고리 골격 9 → 36개(집·직장·통근·가족·건강·기기·차·구독·저축·학습·미디어·기념일 …), 값 풀 평균 6 → 11 → 카테고리-세트 조합 336 → 1.35×10⁸. 과거대화 요약도 6 → 28종이며 이름·숫자·날짜를 담아 "이전 대화의 매우 상세한 것"까지 정밀 회수 가능.
  • Persona: 성격 골격(trait×role×focus) 800 → 10,080 × 음절 생성 에이전트 이름(무한) × 풍부해진 유저 프로필.

→ 결론: MCP는 공식 76k 커버리지가 상한(정상), Memory/Persona는 무한에 가까워 원하면 지금 비율(각 11.9%)보다 더 큰 몫으로 안전하게 확장 가능. 균형을 mem/persona 쪽으로 기울여도 다양성 손실 없음.

현재 양산: EN-only 50k (풀의 ~3.6% 표본). BASE 비율 유지 · 시드 무재사용. 표는 gen30 BASE(42 EN 슬롯) 실측 비율 기준.
